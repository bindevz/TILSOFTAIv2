You are the implementation agent for the repository TILSOFTAIv2.

You are executing Sprint 8.

Baseline repository reality (after Sprint 7):
- `LegacyChatDomainAgent` was deleted.
- `GeneralChatAgent` now owns unclassified/general requests and only uses legacy fallback explicitly.
- Native capability execution is policy-gated through `RequiredRoles` and `AllowedTenants`.
- `RestJsonToolAdapter` supports config-driven binding, timeout, retry, auth/header metadata, tenant/correlation propagation, and classified failures.
- `/health/ready` uses `NativeRuntimeHealthCheck`, while module health is legacy diagnostic only.
- Unit and integration suites are green, with one intentionally bounded skipped deep analytics E2E test.
- However, the platform is still not fully enterprise-grade because bridge/module legacy infrastructure still exists, secret sourcing is not yet platform-governed, capability contracts are still light, readiness is still partly domain-hardcoded, and SQL remains the dominant production path.

Sprint 8 mission:
Turn the platform from a bounded enterprise-grade core into a more fully governed enterprise-grade runtime.

Primary objective:
Shrink legacy bridge dependence again, extract module-era ownership from the runtime story, govern capability sourcing and secrets properly, harden capability contracts, generalize readiness, and prove a second governed non-SQL external path.

Strategic success condition:
After Sprint 8, the platform should look less like:
“native runtime plus carefully bounded legacy residue”
and more like:
“a governed native platform with explicitly temporary compatibility residue”.

Architecture principles:
1. `ISupervisorRuntime` remains the orchestration owner.
2. Domain agents remain the native execution owners.
3. `GeneralChatAgent` remains bounded and supervisor-native.
4. Capability descriptors are native execution boundaries, but production capability shape must become more durably governed.
5. Adapters remain the integration boundary.
6. ApprovalEngine remains the sole write-governance owner.
7. Legacy bridge and module support must continue shrinking.
8. External integration auth and connection metadata must be platform-governed.
9. Readiness must reflect runtime truth generically, not only current domain assumptions.
10. Enterprise-grade progress must be measured by reduced legacy influence, stronger contracts, better governance, and cleaner operational semantics.

Primary Sprint 8 goals:

A. Extract module-era tool catalog ownership
- Replace module-first legacy tool catalog behavior with capability-pack loading or an equivalent platform-native legacy support mechanism.
- Reduce dependency on:
  - `IModuleLoader`
  - `ModuleLoader`
  - `ModuleLoaderHostedService`
  - `IModuleScopeResolver`
  - `ModuleScopeResolver`
  where safe.
- Make it explicit that module-era services are not part of active native runtime identity.
- Remove dead registrations and dead compatibility paths where possible.

B. Shrink `LegacyChatPipelineBridge` again
- Identify which request classes still reach bridge fallback.
- Remove at least one meaningful bridge-bound request class through native handling.
- Tighten bridge reason reporting and operational measurement.
- Keep fallback bounded and explicitly temporary.

C. Govern production capability sourcing
- Move production capability definitions away from reliance on static code literals where possible.
- Improve durable capability source support through configuration, connection catalog, capability source abstractions, or equivalent governed platform data.
- Make precedence explicit between:
  - static defaults
  - environment/app configuration
  - durable platform capability sources
- Add tests for precedence and override behavior.

D. Introduce production-grade secret/connection governance for external integrations
- Do not treat auth token or API key values as ordinary metadata.
- Source REST auth/header material from a platform secret provider or governed external connection catalog.
- Ensure logs/telemetry do not leak secret-bearing values.
- Add tests for secret-backed external capability execution configuration.

E. Add contract-first validation for representative capabilities
- Introduce schema validation or strong request-contract validation for:
  - at least one warehouse SQL capability
  - at least one accounting SQL capability
  - the REST stock capability
- Validation must happen before adapter execution.
- Validation failures must be deterministic and operator-friendly.
- Add tests for invalid argument shapes.

F. Generalize native readiness
- Evolve `NativeRuntimeHealthCheck` beyond hardcoded warehouse/accounting assumptions.
- Readiness should validate the generic native runtime by loaded capabilities, required adapters, and critical orchestration dependencies.
- Keep separate health semantics for:
  - native runtime
  - legacy diagnostics
  - external dependencies
- Update docs and runbooks accordingly.

G. Prove a second governed non-SQL production-style path
- Add one more governed external capability path.
- Prefer a second REST integration or a gRPC-backed capability.
- Include:
  - governed connection metadata
  - secret sourcing
  - timeout/retry/failure handling
  - observability
  - tests

H. Deepen policy beyond role/tenant-only checks where feasible
- Add optional provider-driven system policy checks or equivalent extensibility.
- Preserve the current `RequiredRoles` / `AllowedTenants` gates.
- Add tests for policy denial reasons and observability.

I. Reduce bounded skipped deep analytics validation if practical
- Either unskip the remaining deep analytics E2E test,
- or move it into a clearly bounded external workflow validation suite with explicit ownership and execution rules.
