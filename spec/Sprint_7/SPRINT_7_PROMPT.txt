You are the implementation agent for the repository TILSOFTAIv2.

You are executing Sprint 7.

Baseline repository reality (after Sprint 6):
- Edge entrypoints now call `ISupervisorRuntime` directly.
- `IOrchestrationEngine`, `OrchestrationEngine`, and `ActionApprovalService` were deleted.
- Warehouse and Accounting both have native SQL-backed capability paths.
- Warehouse also has one REST/JSON-backed capability proof path.
- Runtime execution telemetry now exists for supervisor, native, bridge, approval, capability invocation, and adapter failure paths.
- Compatibility debt is explicitly documented.
- However, the repository is still not fully enterprise-grade because legacy fallback and module-era infrastructure still retain too much influence, and the validation baseline is not yet strong enough to support a full enterprise-grade claim.

Sprint 7 mission:
Move the platform from enterprise-ready foundation into a more credible enterprise-grade operating core.

Sprint 7 must prove all of the following:
1. Catch-all fallback is no longer owned by `LegacyChatDomainAgent` in its current form.
2. Bridge fallback is smaller, more explicit, and more operationally diagnosable.
3. Module-era runtime influence is reduced again.
4. Non-SQL adapter support is productionized rather than left as proof-only behavior.
5. Validation confidence improves materially.
6. Capability execution policies become stricter and more testable.
7. The repository becomes cleaner and more operationally trustworthy.

Primary objective:
Reduce legacy fallback dependence, productionize external adapter behavior, and improve the trustworthiness of the runtime and validation baseline.

Strategic success condition:
After Sprint 7, the platform should no longer look like “native runtime plus a large legacy catch-all safety net”.
It should look like a supervisor-native platform with bounded legacy residue and governed external integrations.

Target architecture principles:
1. SupervisorRuntime owns orchestration.
2. Domain agents own domain execution.
3. A general/chat agent, if needed, must also be supervisor-native.
4. Capability contracts remain the native execution boundary.
5. Tool adapters remain the integration boundary.
6. ApprovalEngine remains the only write governance owner.
7. Legacy fallback must be exceptional, measured, and shrinking.
8. Module-era infrastructure must not define the platform’s active identity.
9. Enterprise-grade progress must be validated by real deletions, real policy enforcement, and stronger test credibility.

Primary Sprint 7 goals:

A. Replace or reduce LegacyChatDomainAgent
- Introduce a supervisor-native general/chat agent or equivalent bounded replacement.
- Move catch-all conversational handling away from the current legacy catch-all agent pattern.
- Do not let the replacement simply proxy blindly to LegacyChatPipelineBridge.
- Define explicit fallback ownership for:
  - unsupported requests
  - conversational general requests
  - cross-domain requests
- Delete LegacyChatDomainAgent if safe.
- If deletion is not yet safe, reduce it to the smallest possible compatibility shim with explicit removal conditions.

B. Reduce and classify bridge fallback
- Improve fallback reason classification.
- Make bridge fallback operationally diagnosable.
- Reduce the set of requests that reach bridge fallback.
- Add counters/logging/telemetry details that distinguish:
  - no domain hint
  - no capability match
  - no adapter available
  - general-chat fallback
  - explicit legacy fallback
- Do not let bridge fallback silently hide architecture gaps.

C. Extract module-era tool discovery from the runtime story
- Replace module-first legacy tool discovery with a capability-pack style legacy support mechanism or a similarly bounded alternative.
- Make it explicit that module loading is legacy-only support if it still remains.
- Reduce active dependency on:
  - IModuleLoader
  - ModuleLoader
  - ModuleLoaderHostedService
  - IModuleScopeResolver
  - ModuleScopeResolver
  where safe.
- Health and readiness must clearly separate native runtime readiness from module-era fallback diagnostics.

D. Productionize the REST/JSON adapter path
- Move REST endpoint binding fully into configuration or connection metadata rather than proof-only hardcoded binding assumptions.
- Add external integration policy support:
  - timeout
  - retry or resilience policy
  - optional auth/header policy
  - stronger failure classification
- Preserve tenant and correlation propagation.
- Add tests that validate realistic external adapter behavior, not just happy-path request capture.

E. Improve validation baseline credibility
- Fix or isolate known failing analytics and renderer tests.
- Address IdentityModel dependency/version mismatch or isolate affected HTTP integration tests cleanly.
- Produce a documented “green baseline” for the repository.
- Do not leave known suite failures unaccounted for.

F. Strengthen capability-level authorization/policy enforcement
- Add capability execution denial paths by tenant and/or role policy.
- Ensure unauthorized capability invocation fails deterministically.
- Add tests for denied execution.
- Keep write-path governance unchanged or stronger.

G. Documentation and readiness alignment
- Update README to Sprint 7 truth.
- Update architecture docs.
- Update compatibility debt report.
- Update enterprise readiness gap report.
- Add or update a readiness note that separates:
  - native runtime readiness
  - legacy fallback readiness
  - external integration readiness
