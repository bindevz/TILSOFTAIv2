Sprint 12 mission:
Turn the platform from “production-safe by design” into “enterprise-grade by demonstrated operation”.

Primary objective:
1. Run live certification of the control plane in staging / prod-like environments.
2. Add deployment and promotion gates that block unsafe catalog states and unsafe catalog changes.
3. Define and instrument control-plane SLOs, alerts, and escalation paths.
4. Improve audit and evidence ergonomics for high-risk changes and rollback chains.
5. Enforce contract governance as a release discipline.
6. Tighten fallback and break-glass containment.
7. Produce signed-off operational evidence.

Architecture principles:
1. `ISupervisorRuntime` remains the runtime orchestration owner.
2. Platform catalog records remain the intended production source-of-truth.
3. Bootstrap configuration remains fallback only, not the normal control plane.
4. The governed mutation lifecycle must not weaken.
5. Operational evidence is now the main enterprise-grade differentiator.
6. Promotion gates must become first-class safety controls.
7. Observability must support operator action, not just data exhaust.
8. Contract lifecycle must be part of change governance, not optional metadata.
9. Break-glass and fallback must remain rare, explicit, and reviewable.
10. Sprint 12 success is measured by production trust and certification evidence, not runtime novelty.

Primary Sprint 12 goals:

A. Live-certify the control plane
- Execute the catalog runbook in staging / prod-like environments.
- Execute failure drills for:
  - preview failure
  - version conflict
  - duplicate submit
  - SQL unavailable during apply
  - fallback risk in production-like posture
- Capture operator evidence and sign-off.
- Make certification artifacts inspectable.

B. Harden deployment and promotion gates
- Block promotion when source mode is unsafe.
- Block promotion when preview fails.
- Block promotion when contract validation fails.
- Block production apply when approvals, expected versions, or policy requirements are missing.
- Make the release path self-defending.

C. Add control-plane SLOs, SLIs, alerts, and escalation
- Define preview / submit / approve / apply / rollback service levels.
- Alert on:
  - production fallback source modes
  - repeated version conflicts
  - repeated duplicate submits
  - apply failures
  - rollback surges
- Define escalation ownership and runbooks.

D. Improve audit and evidence ergonomics
- Link change tickets, change ids, correlation ids, and rollback chains more clearly.
- Make high-risk change history easier to inspect.
- Improve incident and post-incident reviewability.

E. Enforce contract governance as release policy
- Require preview for all production-bound catalog changes.
- Require contract version hygiene.
- Define breaking and non-breaking contract handling.
- Add governance around schema artifacts when `SchemaRef` is used.
- Preserve deterministic operator-readable validation behavior.

F. Tighten emergency path containment
- Define who may authorize fallback re-enable in emergencies.
- Define who may invoke break-glass and how it is audited.
- Require after-action review for any emergency path use.
- Ensure emergency paths cannot quietly become normal paths.

G. Continue enterprise proof, not feature expansion
- Only add new runtime surface area if it directly improves certification, governance, or operational proof.
- Do not turn Sprint 12 into a feature sprint.
