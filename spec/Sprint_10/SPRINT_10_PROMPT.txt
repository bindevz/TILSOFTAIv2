Sprint 10 mission:
Complete the platform control plane so the system is not only safe to run, but safe to change under enterprise operating conditions.

Primary objective:
1. Implement governed write paths for capability and external connection catalogs.
2. Add approval/review and audit behavior for catalog mutation.
3. Make bootstrap fallback visible, bounded, and operationally explicit.
4. Expand typed contracts from representative to broader production coverage.
5. Reduce or reclassify residual module package compatibility.
6. Improve catalog observability and integrity validation.

Architecture principles:
1. `ISupervisorRuntime` remains the runtime orchestration owner.
2. Platform catalog records are the intended production source-of-truth.
3. Bootstrap configuration is fallback, not the primary control plane.
4. Catalog mutation must be governed, auditable, and role-controlled.
5. Adapters remain the execution boundary.
6. ApprovalEngine remains the only write-governance owner for business writes, and catalog mutation must have equivalent governance discipline.
7. Typed contract validation must become systemic, not selective.
8. Module packages must continue moving out of the runtime identity.
9. Observability must reflect source-of-truth state honestly.
10. Enterprise-grade progress must now be measured by control-plane maturity, not runtime novelty.

Primary Sprint 10 goals:

A. Implement catalog write/control-plane paths
- Add governed mutation flows for:
  - capability records
  - external connection records
- Target the SQL-backed catalog shape introduced in Sprint 9.
- Support create/update/disable/list behavior.
- Keep secret values out of stored records; only secret references are allowed.
- Add tests for mutation, persistence, and load behavior.

B. Add governance/approval to catalog mutation
- Catalog writes must not be direct ungoverned persistence calls.
- Add approval/review or equivalent controlled mutation flow.
- Capture change metadata such as:
  - owner
  - reviewer/approver
  - version or change note
  - timestamp
- Enforce role/policy boundaries for catalog mutation.
- Document governance rules clearly.

C. Harden bootstrap fallback visibility
- Emit explicit startup/reporting signals when the system is using:
  - durable platform catalog
  - bootstrap configuration fallback
  - mixed-source mode
- Add readiness/degraded-state semantics or equivalent operational signals where appropriate.
- Prevent bootstrap fallback from silently becoming de facto production source-of-truth.

D. Expand typed contracts across production capabilities
- Extend typed contract coverage beyond representative samples.
- Standardize contract structure and validation behavior across production capability records.
- Ensure SQL and REST capabilities have appropriate contract semantics.
- Keep all validation deterministic and operator-safe.
- Validation must still happen before adapter execution.

E. Reduce or reclassify module packages
- Identify which remaining module packages are:
  - still runtime-relevant
  - packaging-only
  - diagnostic-only
- Migrate eligible metadata to platform catalog/tool records.
- Reduce runtime ambiguity around module packages.
- Document the end-state plan for what remains.

F. Improve catalog observability and integrity validation
- Add load-time validation for:
  - duplicate keys
  - malformed records
  - unresolved external connections
  - secret reference misuse
  - invalid typed contracts
- Add logs/metrics/reporting for source-of-truth usage and mutation events.
