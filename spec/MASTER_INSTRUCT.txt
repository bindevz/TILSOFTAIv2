You are the implementation agent for the repository TILSOFTAIv2.

Mission:
Upgrade the repository from the unfinished v2 architecture to the v3 architecture.

Repository reality:
- The current repository is a C# orchestration platform with strong SQL-centric execution.
- The current runtime is centered around a single orchestration engine.
- The current codebase still contains module-centric loading and tool registration patterns.
- An approval flow already exists and must be evolved, not discarded.
- The repository is under active development and is not yet fully hardened.

Target v3 architecture:
- SupervisorRuntime
- Domain Agents
- Tool Adapters
- Approval Engine
- ERP-agnostic integration through SQL, REST API, Queue, File Import, and Webhook
- Approval-first governance for every write intent
- Multi-tenant safety by design
- Enterprise internal AI platform design

Primary objective:
Transform the repository into v3 architecture with minimal unnecessary churn, maximum reuse of valid assets, and aggressive removal of obsolete abstractions, dead code, and legacy patterns that do not fit v3.

Non-negotiable rules:
1. Do not add architecture that duplicates existing valid capabilities.
2. Do not keep legacy abstractions only for backward comfort.
3. Do not rename components unless the rename supports the target architecture.
4. Do not leave parallel competing runtimes in place without explicit justification.
5. Do not preserve module-centric runtime behavior as the main architecture.
6. Do not hardcode domain logic in API or infrastructure layers.
7. Do not couple domain agents directly to SQL Server specifics.
8. Do not allow write execution without passing through ApprovalEngine.
9. Keep the platform ERP-agnostic at the integration boundary.
10. Preserve tenant isolation and policy enforcement at every layer.

What must be preserved and evolved:
- execution context
- conversation persistence
- observability foundations
- resilience patterns
- streaming support
- existing approval flow concepts
- existing schema validation concepts
- useful SQL execution capabilities, but only as one adapter type

What must be reduced or removed:
- module-centric runtime as the primary execution model
- reflection-heavy module loading as a central design
- SQL-first assumptions in domain orchestration
- duplicate orchestration layers
- dead services, dead interfaces, dead options, dead handlers
- stale README and outdated architectural claims
- any configuration or documentation that no longer matches runtime reality

Transformation principles:
- module-centric runtime must stop being the primary runtime model
- agents must own domain runtime behavior
- adapters must own integration behavior
- approval engine must own write execution governance
- SQL must become one adapter type, not the architectural center
- obsolete code must be deleted when safe
- duplicated runtime paths must not be kept without explicit justification

Required working style:
- inspect the current code before changing it
- tie every change to a concrete existing file or abstraction
- prefer refactoring over layering more wrappers
- prefer deletion over indefinite coexistence
- produce exact file-level and abstraction-level change reports
- keep changes incremental and reviewable

Execution protocol:
Phase 1:
- analyze current repo structure
- map v2 components to v3 target components
- identify keep / evolve / delete / replace items

Phase 2:
- define the v3 runtime boundaries
- introduce SupervisorRuntime, AgentRegistry, DomainAgent abstractions, ToolAdapter abstractions, ApprovalEngine abstractions
- keep backward-compatible facades only where absolutely necessary

Phase 3:
- migrate current orchestration to Supervisor + Domain Agent flow

Phase 4:
- transform SQL execution into SqlToolAdapter
- prepare REST / FileImport / Queue / Webhook adapter contracts

Phase 5:
- evolve current action approval flow into a reusable ApprovalEngine
- ensure every write intent produces a draft or approval-required action first

Phase 6:
- remove obsolete code
- remove dead abstractions
- remove legacy runtime paths that are no longer part of v3
- update documentation
