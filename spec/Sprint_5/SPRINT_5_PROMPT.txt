You are the implementation agent for the repository TILSOFTAIv2.

You are executing Sprint 5.

Baseline repository reality (after Sprint 4):
- SupervisorRuntime is real and already classifies, routes, and executes through agents.
- WarehouseAgent has the first real native domain execution path through capability resolution and ToolAdapterRegistry.
- AccountingAgent still exists, but remains bridge-dependent and does not yet own a native capability path.
- LegacyChatPipelineBridge still exists as a compatibility fallback for unsupported or non-native domain paths.
- ApprovalEngine and IWriteActionGuard already govern write execution.
- CapabilityDescriptor, ICapabilityRegistry, InMemoryCapabilityRegistry, and WarehouseCapabilities now exist.
- Chat controller edge auth has been hardened compared with earlier sprints, and the runtime now uses authentication and authorization middleware.
- The platform is no longer only an architectural shell, but it is still transitional and still depends too much on compatibility paths.

Sprint 5 mission:
Convert Sprint 4’s single native-domain proof into a repeatable platform pattern.

Sprint 5 must prove all of the following:
1. Native execution is not limited to one warehouse demo path.
2. Capability resolution is more reliable than naive keyword matching.
3. Capability registration can become data-driven instead of static-only.
4. Critical runtime paths are validated with realistic integration tests.
5. Compatibility debt is shrinking, not growing.

Primary objective:
Implement a second real native domain path, preferably in AccountingAgent.

Sprint 5 success condition:
The repository must no longer look like “Warehouse is native, everything else still bridges”.
By the end of this sprint, the platform must support at least two real native read paths across two domains.

Target architecture principles:
1. SupervisorRuntime remains the orchestration owner.
2. Domain agents own domain runtime decisions.
3. Capability contracts remain the execution boundary for native paths.
4. Tool adapters remain the integration boundary.
5. ApprovalEngine remains the only write governance owner.
6. LegacyChatPipelineBridge remains fallback-only and must continue losing architectural weight.
7. SQL is still allowed as an adapter implementation, but not as a domain orchestration center.
8. Sprint 5 must improve execution quality, not just add more abstractions.

Primary Sprint 5 goals:

A. Second native domain path
- Implement at least one real native read-only capability for AccountingAgent.
- Recommended capability candidates:
  - accounting.receivables.summary
  - accounting.payables.summary
  - accounting.invoice.by-number
- AccountingAgent must execute the capability through:
  - capability resolution
  - ToolAdapterRegistry
  - ToolExecutionRequest
  - adapter execution
- AccountingAgent must not call SqlExecutor directly.
- LegacyChatPipelineBridge may remain fallback-only when no native accounting capability applies.

B. Better capability resolution
- Replace or reduce naive string-contains matching as the main capability selection mechanism.
- Introduce a more reliable capability selection strategy.
- Acceptable implementation patterns:
  1. structured capability hint supplied by SupervisorRuntime
  2. per-domain ICapabilityResolver
  3. explicit task metadata that maps to capability keys
- Unacceptable long-term pattern:
  - continuing to depend mainly on matching fragments of capability key text in freeform input
- Sprint 5 must migrate at least Warehouse and Accounting to the improved capability resolution path.

C. Data-driven capability registry
- Reduce dependence on purely static hardcoded capability lists.
- Introduce a data-driven capability source.
- Acceptable first step:
  - configuration-backed capability loading
  - SQL-backed capability loading
  - composable registry with startup loading from data source
- InMemoryCapabilityRegistry may remain only as:
  - fallback
  - test fixture
  - bootstrap support
- Capability metadata must remain explicit and must still include:
  - capability key
  - domain
  - adapter type
  - operation
  - target system id
  - integration binding
  - execution mode
  - optional policy / argument metadata if needed

D. Integration validation
- Add infrastructure-backed or realistic runtime integration tests for:
  1. Warehouse native path
  2. Accounting native path
  3. Approval lifecycle with realistic persistence boundary
  4. Auth-enabled request path
- Do not rely only on mocked unit tests for Sprint 5 success.
- Integration tests do not need full production infra, but they must validate real wiring and real path behavior.

E. Compatibility debt reduction
- LegacyChatPipelineBridge must not gain new architectural responsibilities.
- OrchestrationEngine must remain compatibility-oriented only.
- Do not add new domain behavior to compatibility shims.
- Every remaining compatibility component must be explicitly marked with:
  - why it still exists
  - what depends on it
  - what removes it later
- If safe, reduce or simplify compatibility paths as part of this sprint.

F. Documentation alignment
- Update README so it reflects Sprint 5 runtime truth.
- Update architecture notes if the runtime shape changes materially.
- Document:
  - Accounting native path
  - improved capability resolution
  - data-driven capability registry step
  - remaining compatibility debt
  - known limitations
  - Sprint 6 priorities
